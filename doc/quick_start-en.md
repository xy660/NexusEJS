# NexusEJS Quick Guide

## Build & Deploy

### Write and Compile Scripts

1. Write your JavaScript script (experienced JS developers can skip to the syntax examples below for a quick start).
2. Run the `nejsc` compiler command: `nejsc your_app.js`. This generates `your_app.nejs` and its source map file `your_app.nejs.map` in the current directory.

### Compile Firmware and Deploy

1. Select the VM version matching your MCU and build the PlatformIO project. Prepare the SPIFFS filesystem.
2. Rename the `.nejs` file generated by NexusEJS-Compiler (`nejsc`) to `entry.nejs` and flash it into the filesystem.
3. Flash both the firmware and the filesystem, then reboot the MCU.

### Debugging

1. **Basic Stack Trace Line Number Resolution**
   Copy the `stackTrace` field from the exception object. Load the map file using the `offset2line` tool by running: `offset2line your_app.nejs.map`. Paste the stack trace information when prompted to obtain the corresponding source code line numbers.

2. **Using the nejsdbg Debugger**
   Launch the `nejsdbg` debugger with the following command:
   ```bash
   nejsdbg <mode> <addr>
   ```
   Example:
   ```bash
   nejsdbg serial COM3
   ```

   You will see an interface similar to this:

   ```bash
   Connected COM3
   Restart device to trigger the default breakpoint.
   ====BreakPoint====
   Stack trace:
     at main_entry() line:1
   Scopes:
     {start:0, length:325, ep:325, attr:0}
     {start:6, length:319, ep:0, attr:0}
   NexusEJS.Debugger>>
   ```

   If the breakpoint does not trigger automatically, press the device's RST button to reboot. The VM enabled for debugging will automatically break at the first line of the `main_entry` function.

---

NexusEJS syntax and runtime are optimized for embedded systems; therefore, memory management and coding style differ from web JavaScript.

## Variable Declaration

The NexusEJS compiler disables `var`. You must use `let` / `const` to define variables. While the compiler checks `const`, it currently does not enforce read-only behavior.

## Object-Oriented Programming

NexusEJS does not currently support the `class` keyword. However, constructors and prototype chains remain available (similar to ES5 style). You should use constructors or the factory pattern to create objects.

```javascript
// Constructor

function Human(name, age) {
  this.name = name;
  this.age = age;
}
Human.prototype.getInfo = function () {
  return "My name is " + this.name + " and my age is " + this.age;
};

let man = new Human("alice", 25);

// Factory Pattern
global.Human = {
  create(name, age) {
    return {
      name: name,
      age: age,
      getInfo() {
        return "My name is " + this.name + " and my age is " + this.age;
      },
    };
  },
};

let man = Human.create("bob", 25);
println(man.getInfo());
```

Here, `global` represents the global namespace, similar to `window` in browsers. Using the factory pattern avoids the memory overhead of prototype chains. However, if you have many objects sharing methods, using constructors with prototype chains is the better choice.

**Note:** To save memory, the following types are treated as objects but **do not support adding members or modifying prototypes**:
- Array
- String

The following types are **not** objects. They become special objects only when boxed, have no prototype, and cannot be accessed using the dot operator:
- number
- boolean
- null
- undefined

These are temporary implementations and may change in the future.

## Closures

NexusEJS supports closures. A closure is triggered when a function:
- Is passed as an argument
- Is returned as a value
- Is assigned to a variable

**Use closures judiciously to avoid unnecessary memory consumption.** A closure is only created when a lambda expression or function references variables outside its immediate scope.

**Primitive values captured by closures are automatically boxed into objects, consuming additional heap memory.**

Example:
```javascript
function getCounter() {
  let cnt = 0;
  let obj = { value: 0 };
  return () => {
    cnt++;
    obj.value++;
  };
}

let inc = getCounter();
inc();
```

## Comparison Operators

NexusEJS runtime behavior defaults to a stricter mode similar to `"use strict"` in traditional JavaScript.

NexusEJS does not support mixing `"123"` and `123`. The `==` operator behaves identically to the `===` operator in standard JavaScript. You must convert types explicitly using `Number.parseInt()` or `Number.parseFloat()`.

Similarly, array indices do not allow strings to prevent undefined behavior caused by implicit conversion.

## Type Conversion

NexusEJS uses strict typing (`"123" != 123`), requiring explicit type conversion.
- **To String:** Explicit `toString()` calls are neither needed nor supported. NexusEJS auto-converts during string concatenation (e.g., `123 + "" == "123"`).
- **String to Number:** Use `Number.parseFloat(str: string)` (e.g., `Number.parseFloat("123") == 123`).

Arrays do not support numeric indices represented as strings. Usage like `❌ arr["123"]` will throw an exception immediately.

## Memory Management

NexusEJS uses automatic Garbage Collection (GC), which triggers when free memory falls below a threshold defined by the `LESS_MEMORY_TRIG_COLLECT` macro in `GC.h`.

You can also manually trigger GC via the native binding:
```javascript
gc();
```

GC triggers a "Stop The World" (STW) pause. Use manual GC strategically—for example, before high-priority real-time tasks—to avoid unexpected latency.

### READY TO FINALIZE Mechanism

The READY TO FINALIZE mechanism allows objects to control their lifecycle and release native resources by implementing a `finalize(): boolean` method.

When GC is about to reclaim an object, it calls `finalize()`. Returning `false` prevents the GC from collecting the object and its referenced children. Returning `true` removes the `finalize` method, allowing collection on the next GC cycle.

**Warning: Do not call `obj.finalize()` directly. This method is reserved for GC callbacks only and will result in undefined behavior.**

## Multitasking

NexusEJS differs from standard JavaScript by removing the event loop in favor of an M:N virtual thread scheduler. One OS Task (thread) schedules multiple Virtual Threads (VTs).

**This architecture allows developers to write synchronous-style code using polling loops without freezing the entire VM (though using `vtDelay(0)` to yield is recommended for better system responsiveness).**

The `async` and `await` keywords have been removed. Use `runTask(func, param): TaskObject` to create tasks and obtain a `TaskObject`:
- `.id`: Get the task ID.
- `.isRunning()`: Check if the task is active.
- `.getResult()`: Get the return value (returns `null` if still running).

For concurrent resource access, use OS-level mutexes or critical sections. NexusEJS provides the `lock(obj){...}` syntactic sugar:

```javascript
let sharedObject = {...};

// Access in another task
lock(sharedObject) {
    // Operation is safe here
}
```

To manage concurrency between two VTs within the same thread, use:
```javascript
vtSetScheduleEnabled(true | false);
```
This pauses the VT scheduler's preemption mechanism, effective only for VTs within the current physical/OS thread.

**Default Task Stack Size:** 8KB (unavailable in environments without an OS).

For lightweight threading within the current OS thread, use Virtual Threads:
```javascript
vtStart(() => {...});
vtStart(myWork);
```
VTs run via the NexusVT scheduler using time-slicing. Do not use `delay` or `System.delay`; use `vtDelay` instead. Long blocking I/O in the native layer within a VT may starve other VTs of CPU time; use `runTask` in RTOS environments for such operations.

**Caution:** Do not use the `lock` statement between two VTs sharing the same thread, as this will cause a deadlock.

## Buffer Type

The `Buffer` type is a native-bound class designed specifically for NexusEJS, allowing scripts to allocate native memory buffers of a specific size.

Usage Example:
```javascript
let size = 32;
let buf = Buffer.create(size);
let offset = 0;

// writeUInt(offset, size, value)
buf.writeUInt(offset, 4, 123); // size range: 1-8. 64-bit integers may overflow.

// readUInt(offset, size)
let read = buf.readUInt(offset, 4);

// write string
buf.writeUTF16(offset, "hello");

buf = null; // Dereference
gc();       // Call GC to free the native buffer
```
